# Phase 5: Search & Book

<objective>
To officially enable the Booking cycle logic. Most of the geospatial Socket pipeline is actually already scaffolded implicitly in `bookingController.js`, so our focus is verifying these complex WebSocket handshakes, and injecting the missing `cancelBooking` REST method so Users can cleanly abort "pending" reservations.
</objective>

<context>
The existing architecture natively hooks `createBooking`, `acceptBooking`, `startTrip`, and `endTrip` over Socket.io `io.to()` broadcasts targeting individual `userId` rooms. However, there is no way for a Tourist to abort a pending or accidentally accepted booking, which leaves Guides permanently blocked (`guide.currentBooking` is not null). We need to create a `/cancel/:id` route to unlock systems.
</context>

<task>
<read_first>
- `backend/controllers/bookingController.js`
- `backend/routes/bookingRoutes.js`
</read_first>
<action>
1. Edit `backend/controllers/bookingController.js`. At the EOF, add `cancelBooking` method taking a URL param `:id`.
2. Find the Booking. If it exists and status is `pending` or `accepted`, set it to `cancelled`.
3. If `booking.guideId` exists, look up the Guide and set `currentBooking: null`. Then `io.to(guide.userId.toString()).emit('bookingCancelled', { bookingId: booking._id })`.
4. Export `cancelBooking`.
5. Update `bookingRoutes.js` to map `router.post('/cancel/:id', authorize('user'), cancelBooking)`.
6. Write a `testPhase5.js` Socket+HTTP simulation script to generate a booking, have the fake Guide receive the emission, have the Tourist cancel it, and prove the sockets update gracefully.
</action>
<acceptance_criteria>
- `cancelBooking` successfully frees the Guide model lock.
- Socket routes `newBookingRequest` to Guide when User calls `/create`.
- Socket routes `bookingCancelled` to Guide when User calls `/cancel`.
</acceptance_criteria>
</task>
