# Phase 6: Razorpay Payments

<objective>
To implement a robust transaction system securely connecting Mongoose Bookings to Razorpay via REST. We will model commission math, generate dynamic orders, and implement a signature-validating Webhook to autonomously settle payments without frontend intervention.
</objective>

<context>
The existing stack has no `razorpay` dependencies installed, and the `Booking` schema is unaware of payment structures like `isPaid` or internal platform dividends. By building the `paymentController.js`, we can intercept finalized trips and generate active Razorpay Orders natively.
</context>

<task>
<read_first>
- `backend/server.js`
- `backend/models/Booking.js`
</read_first>
<action>
1. Execute `npm install razorpay --save` inside the `backend` environment.
2. Modify `backend/models/Booking.js` to include `{ isPaid: Boolean, paymentId: String, adminCommission: Number, guidePayout: Number }` defaulting to 0/false.
3. Scaffold `backend/controllers/paymentController.js` with `createOrder()` isolating the Razorpay SDK initialization using `.env` keys. Calculate `20%` admin commission.
4. Scaffold `backend/controllers/paymentController.js` with `verifyPayment()` implementing `crypto` signature hatching against the `razorpay_signature`. If valid, hit Mongo and flip `isPaid = true`.
5. Scaffold `backend/routes/paymentRoutes.js` and patch it into `backend/server.js` (`/api/payments`).
6. Validate through a mocked script bypassing Razorpay's actual sandbox by simulating the `.env` crypto hatch.
</action>
<acceptance_criteria>
- `razorpay` installed.
- `Booking.js` supports transaction mechanics.
- `paymentController` hosts Webhooks & Generators.
- `server.js` actively parses `/api/payments`.
</acceptance_criteria>
</task>
