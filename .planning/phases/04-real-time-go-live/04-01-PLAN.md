# Phase 4: Real-time & Go Live

<objective>
To fully resolve real-time WebSocket capabilities across the application. When a guide updates their location via REST, it needs to broadcast out to all listening tourist map clients. When a guide disconnects their WebSocket cleanly or uncleanly, it needs to signal an offline status broadcast.
</objective>

<context>
`utils/socket.js` and `guideController.js` have the basic boilerplate for sockets and updating MongoDB respectively. However, `updateLocation` REST endpoint does not emit the `io.emit` broadcast required for active browsers to see moving icons. Additionally, the `disconnect` event on `utils/socket.js` updates the DB to `isOnline: false` but forgets to tell the clients via a socket emission.
</context>

<task>
<read_first>
- `backend/utils/socket.js`
- `backend/controllers/guideController.js`
</read_first>
<action>
1. Modify `backend/controllers/guideController.js` inside the `updateLocation` method. Check if `isOnline` is true (or just globally emit). Upon successful MongoDB update, import `../utils/socket.js`, `getIO()`, and `io.emit('guideLocationUpdate', { guideId: guide._id, lat, lng })`.
2. Modify `backend/utils/socket.js` inside the `disconnect` handler. When `findOneAndUpdate` sets `isOnline: false`, immediately execute `io.emit('guideStatusUpdate', { guideId: guide._id, isOnline: false })`.
3. Create a unified Node test script `testPhase4.js` utilizing `socket.io-client` dependency locally to ping the backend socket instance, prove connection is accepted, and wait for a broadcast emission.
</action>
<acceptance_criteria>
- `guideController.updateLocation` emits socket update.
- `utils/socket.js` disconnect handler emits offline broadcast.
- CLI tests successfully connect via WebSocket.
</acceptance_criteria>
</task>
